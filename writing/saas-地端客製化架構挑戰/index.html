    我如何處理 SaaS 與地端客製化並行的架構挑戰¶

    我之前遇過一個蠻有挑戰性的問題，是在維護「SaaS + 地端客製化」並行的產品架構。

    這類問題難的地方，不只是某個功能怎麼寫，而是在架構、維護成本、商業需求和團隊資源之間，如何做取捨。

    背景：SaaS 與地端版本同時存在¶

    當時產品同時有 SaaS 版跟地端版。

    SaaS 是一個獨立專案，而地端則需要支援不同客戶的客製化需求。因此地端會基於一個核心版，再延伸出不同客戶版本去維護。

    我主要負責地端這邊。

    基於地端人員配置比 SaaS 端更精簡的情況，SaaS 開發速度本來就比地端快，再加上地端本身又有許多客製化功能，久而久之，兩邊的落差就越來越大。

    這些落差不只是在功能上，也包含：

    • 程式碼結構
    • 依賴套件版本
    • 資料流設計
    • API flow
    • workflow 狀態管理
    • 技術架構與開發模式

    而維護地端的困難在於：

    • 客戶仍希望地端能持續同步擁有 SaaS 的許多新功能
    • 地端本身仍持續有很多自己的客製化功能

    所以很多功能不是 copy/paste 就能完成，而是要理解兩邊功能，再分析、拆解、搬運與調整相容性。

    我實際遇到的情況¶

    當時有一個新需求。

    原本系統是以「單一任務」為核心，一份文件或流程就是一個任務。

    但後來有些企業使用情境希望新增一個更高層級的概念，類似「群組」或「集合」：

    可以把多個任務包成一組，一次送出、一次管理。

    這個設計在 SaaS 版裡原本沒有，所以我需要分析、評估如何實作並做取捨。

    第一種做法：盡量沿用 SaaS 既有架構¶

    第一種做法，是盡量沿用 SaaS 的既有架構。

    這樣的好處是：

    • 未來同步 SaaS 功能時，比較不容易越差越遠
    • 兩邊的概念、資料流和維護方式比較接近
    • 團隊比較容易判斷功能差異在哪裡

    但壞處是：

    • 為了貼近現有架構，程式碼可能會變得比較不漂亮
    • 部分客製化需求會需要繞過原本的設計限制
    • 短期開發體驗不一定是最理想的

    第二種做法：重新設計更合理的架構¶

    第二種做法，是直接重新設計一套更合理的架構。

    這樣會比較乾淨，也比較符合長期維護。

    但代價是：

    • 地端跟 SaaS 的架構會更加分叉
    • 未來同步 SaaS 功能的成本可能更高
    • 團隊需要同時理解兩套越來越不同的設計

    最後我跟團隊討論後，選擇的是：

    優先維持與 SaaS 的一致性，再用最小改動去完成客製化。

    因為團隊當時的共識是：

    雖然短期內不夠漂亮，但長期來看，維持兩邊架構接近，比較能控制未來成本。

    這個問題後來怎麼處理？¶

    老實說，在我離開前，這個問題沒有被真正完全解決。

    因為它其實不只是技術問題，也包含：

    • 組織變動
    • 人力不足
    • 管理層 prioritization
    • 重構成本難量化

    所以很多重構提案，即使工程師知道重要，也很難真的排進 roadmap。

    但我們當時其實有持續在推幾個方向。

    1. 推動模組化¶

    我跟其他前端工程師有討論：

    未來應該把功能拆成模組，而不是綁在特定流程頁面裡。

    例如，不同流程不應該各自包整套重複邏輯，而是應該把功能拆成更獨立的模組，例如：

    • 欄位能力
    • workflow feature
    • 簽核或審核能力
    • reviewer 機制
    • 特定流程的狀態管理

    這樣未來 SaaS 跟地端才能更容易共享。

    2. 推動 SaaS 反向吸收地端功能¶

    我也有主動提出：

    如果某些地端客製化功能已經被很多企業使用，其實應該評估整併回 SaaS。

    這樣可以：

    • 降低雙方差異
    • 統一寫法
    • 減少長期維護成本

    例如一些企業常見能力、身分驗證流程、特殊 workflow abstraction，我認為都很適合回收進 SaaS 核心。

    3. CI/CD 改善¶

    另外我也有實際參與一個 CI/CD 優化。

    因為多 repo、多 branch 的情況下，不同分支使用的 dependency 跟 image 都不同，導致 CI 很容易壞掉。

    以前團隊是手動 mapping branch 跟 image。

    後來我推動改成 multiple-stage 的方式，讓 CI 能依 branch 自動吃對應設定與 image。

    這部分在我離開前，前端已經完成，後端也做到一半。

    4. AI 導入的想法¶

    我後期其實最有興趣的，是研究：

    能不能用 AI 去降低「SaaS 功能搬遷到地端」的成本。

    因為那個流程本質上很適合 AI：

    • 比對兩邊 repo
    • 找差異
    • 搬功能
    • 調整相容性
    • 跑測試
    • 驗證影響範圍

    我那時候已經開始研究：

    • Codex
    • Claude Code
    • BDD / TDD workflow
    • AI-assisted migration

    因為以前這件事非常吃人工，而且很容易漏。

    但如果：

    • 測試先定義好
    • Prompt 設計好
    • AI 可以讀整個專案

    其實很多重複性的搬遷與驗證，可以被大幅自動化。

    我認為這未來有機會把開發效率提升非常多。

    我從這件事學到的¶

    這件事情讓我學到：

    很多真正困難的工程問題，不一定是「技術做不到」。

    而是：

    在架構、維護成本、商業需求、團隊資源之間，怎麼做取捨。

    以及：

    如何在不完美的現實下，讓系統不要失控地惡化。

    我覺得這是我在那段經驗裡，學到最重要的事情。
