# async/await 非同步語法?

在程式設計中，`async/await` 被稱為**非同步語法。**

不是因為他使用後會讓程式變成非同步，相反的，他其實是讓該程式的非同步流程變得像同步。

<!-- more -->

那為何他卻被稱為是非同步語法呢?

## 同步這個概念

我們可以先來了解一下同步這個概念，雖然工作上常常會使用到這個用詞，意義上大概是把我們之間對某件事情的理解對齊，消除時間差、信息資訊差。

但是，在前端開發中提到這個詞的時候，我們其實在講的是是否並行執行。

這裡就要先提到 JS thread，我們要知道 JS 是單線程，也就是說，正常情況 JS 程式碼執行時是會一直向下執行下去，中間如果卡住，他就會等待，不會多線程並行。

那但是要處理 API 時，為了不讓 JS 就這樣一直等下去，所以 JS 就用了 Event Loop 的方法，來去做到非同步處理 API 操作。

所以，聰明的你就知道，這裡的同步，意思其實有點像是依序執行，而非同步則是可並行執行（併發）的意思。

## 為何 async/await 被稱為非同步語法

回到為何 `async/await` 被稱為是非同步語法的問題，因為他是用在非同步的流程中，而他在做的其實是定義此為非同步語法，以及操控此非同步語法使其能像同步語法一樣等待。

## 結論

`async/await` 是非同步語法。

它讓你能「用同步的寫法控制非同步流程」。

!!! note "PS1"

    細講的話，`await` 這邊也不是讓他非同步變成同步，只是類似而已。

!!! note "PS2：並發不是平行"

    在計算機科學中，真正的「平行處理（Parallelism）」是指**在同一時間真正同時做好幾件事**（這需要多核心或多執行緒支援）。JS 本身因為是單線程，它一次只能做一件事。
    
    所謂的「非同步」，其實是在單線程下達成**並發（Concurrency）**的手段。它指的是**「我可以先發起一個 API 請求，接著去處理其他任務，等 API 結果回來後再去執行對應的處理」**，這是一種「時間差」的分配藝術，而不是真正的多線程平行。
