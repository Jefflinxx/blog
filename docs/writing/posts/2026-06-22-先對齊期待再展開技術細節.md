# 先對齊期待，再展開技術細節

在團隊協作裡，有些問題表面上看起來是技術問題，但真正卡住的地方，往往不是技術本身。

而是不同角色之間，對「現在要討論的是什麼」、「希望看到什麼成果」、「資訊要用什麼粒度呈現」沒有先對齊。

<!-- more -->

尤其當一個任務開始牽涉到產品方向、系統架構、開發路線和跨角色溝通時，如果我們一開始就直接鑽進實作細節，很容易出現一種情況：

工程師覺得自己已經講得很完整，但其他人仍然覺得難以理解。

這時候問題不一定是內容不夠多，而是內容的層級不對。

## 技術正確，不代表溝通有效

工程師在面對複雜問題時，常見的直覺是：

- 先拆流程
- 先整理系統邊界
- 先畫架構圖
- 先補充技術細節
- 先把想到的風險和限制寫清楚

這些事情本身都沒有錯。

但如果接收資訊的人不是只看實作細節，而是想先理解產品方向、決策邏輯或整體路線，那麼太早進入細節，反而會讓溝通變得更混亂。

因為技術細節通常回答的是：

> 這件事要怎麼做？

但很多跨角色溝通一開始真正需要回答的是：

> 我們現在要解決什麼問題？

> 這個方案的方向是什麼？

> 為什麼要這樣拆？

> 接下來會怎麼推進？

如果這些問題沒有先對齊，後面的流程圖、架構圖、API flow、系統拆分，即使內容是對的，也可能很難被理解。

## 先問清楚對方想看的層級

在開始整理大型任務前，我覺得很重要的一步是：

先確認對方期待看到的是什麼層級的資訊。

例如可以先問：

- 你希望我先整理產品方向，還是直接整理技術方案？
- 這份內容主要是給工程團隊看，還是也會給非技術角色看？
- 你期待看到的是 roadmap、系統架構、功能拆解，還是執行計畫？
- 有沒有過去類似文件或簡報可以參考？
- 這次同步的目標是做決策、收斂方向，還是讓大家知道進度？

這些問題看起來很簡單，但它可以避免一開始就把力氣花在錯的地方。

很多時候，溝通卡住不是因為做得不夠多，而是因為做的東西和對方期待的東西不在同一層。

## 從大方向開始，再往下拆細節

當任務牽涉到多個系統或多個角色時，我會建議不要一開始就從技術細節開始講。

比較好的順序是：

1. 先說明背景與問題
2. 再說明目標與方向
3. 接著說明整體架構或流程
4. 最後才拆到各模組、系統邊界與實作細節

也就是先讓大家知道：

> 我們現在站在哪裡，要往哪裡走，為什麼這樣走。

再進一步說：

> 如果要這樣走，技術上需要怎麼拆。

這樣做的好處是，非技術角色可以先理解方向，技術角色也比較容易接上後面的實作討論。

如果一開始就直接講某個服務怎麼串、某個模組怎麼拆、某個 API flow 怎麼設計，對沒有完整背景的人來說，很容易只看到一堆局部資訊，但不知道這些資訊最後要拼成什麼。

## 文件和簡報，比群組長訊息更適合承載複雜討論

當討論開始變長、變複雜，只靠群組訊息其實很容易失焦。

群組訊息適合快速同步，但不太適合承載需要反覆閱讀、回頭查找、逐步理解的內容。

因為群組訊息有幾個限制：

- 討論脈絡容易被打散
- 資訊順序會被回覆時間影響
- 長訊息閱讀成本很高
- 圖和文字不容易形成穩定結構
- 後續很難回頭找到真正的結論

所以只要內容開始牽涉到產品規劃、架構說明、跨角色決策，我會更傾向先整理成文件或簡報。

文件適合承載完整脈絡，簡報適合做方向同步。

而且簡報不是只為了好看，而是逼自己把內容拆成更容易吸收的單位：

- 單頁只放一個重點
- 文字要大到可以快速掃讀
- 圖像要輔助理解，而不是增加負擔
- 不要把所有細節都塞進同一頁
- 先講結論，再補原因

這些限制反而會讓溝通變清楚。

## 好的技術溝通，是把資訊轉譯給不同角色

我以前會覺得，只要把技術內容寫完整，就是負責任。

但後來慢慢發現，完整不等於有效。

好的技術溝通，不只是把自己知道的東西全部寫出來，而是要根據對象調整呈現方式。

對工程師來說，可能需要看到：

- 系統邊界
- API flow
- 資料流
- 相依關係
- 技術風險
- 實作拆分

但對產品、營運或管理角色來說，他們可能更需要先知道：

- 這個方向解決什麼問題
- 對使用者有什麼價值
- 哪些事情要先做，哪些可以後做
- 風險在哪裡
- 需要什麼資源
- 什麼時候可以看到階段成果

如果把同一份技術文件直接丟給所有角色，常常會有人看不懂，也會有人覺得資訊不夠。

所以更好的方式，是把內容分層：

- 第一層：方向與目標
- 第二層：流程與架構
- 第三層：技術拆解與實作細節
- 第四層：風險、待確認事項與後續行動

這樣每個角色都可以先抓到自己需要的資訊，再往下看細節。

## 遇到「看不懂」時，不要只補更多資料

當別人說「我看不懂」時，工程師很容易直覺反應是補更多資料。

但有時候補更多資料只會讓問題更嚴重。

因為真正的問題可能不是資訊不夠，而是不知道哪裡沒有對齊。

這時候更有效的做法，是先問：

> 你看不懂的是哪一段？

> 是方向不清楚，還是流程不清楚？

> 是不知道為什麼要這樣做，還是不知道接下來要怎麼做？

這些問題可以幫助我們定位問題。

如果對方不理解的是產品方向，那就不該繼續補 API 細節。

如果對方不理解的是角色分工，那就不該繼續補系統架構。

如果對方不理解的是決策原因，那就不該只補流程圖。

先找出沒有對齊的點，再補正確層級的資訊，才會真正解決問題。

## 團隊可以建立的幾個協作習慣

要讓這類溝通變順，不只靠個人表達能力，也可以靠團隊習慣來降低摩擦。

我覺得可以從幾個地方開始：

### 1. 任務開始前先定義輸出格式

在開始做規劃前，先講清楚這次要產出的東西是什麼。

例如：

- 一頁方向簡報
- 一份技術設計文件
- 一張系統流程圖
- 一份 execution plan
- 一份風險與待確認清單

先定義輸出格式，可以避免做完才發現形式不符合期待。

### 2. 每份文件先寫「給誰看」

同一個議題，給不同角色看的版本應該不同。

文件一開始可以先寫：

- 這份文件的讀者是誰
- 這份文件要幫助大家做什麼決策
- 這份文件不包含哪些細節

這可以讓讀者知道要用什麼角度閱讀，也能避免內容無限膨脹。

### 3. 先同步大方向，再同步實作細節

如果方向還沒對齊，就不要急著討論所有細節。

可以先用一份簡短內容確認：

- 問題是否定義正確
- 方向是否合理
- 優先順序是否一致
- 有沒有重大風險被漏掉

方向對齊後，再進入技術拆解，效率會高很多。

### 4. 把「待確認」明確列出來

規劃初期一定會有很多不確定。

不要假裝全部都已經確定，反而應該把不確定的地方列出來。

例如：

- 這個需求是否一定要支援？
- 這個功能是否第一階段就要做？
- 這個系統是否要跟既有服務整合？
- 這個決策會不會影響後續擴充？

把待確認事項列出來，可以讓討論更聚焦，也可以讓團隊知道目前卡點在哪裡。

## 結論

跨角色協作裡，技術能力很重要，但技術溝通同樣重要。

因為很多複雜任務不是只要寫得出來就好，而是要讓團隊理解：

- 為什麼要做
- 要往哪裡走
- 現在要先決定什麼
- 後面會怎麼拆
- 風險和不確定性在哪裡

如果一開始就直接進入技術細節，很容易讓討論卡在局部。

但如果能先對齊期待、建立共同語境，再往下展開架構與實作細節，團隊協作就會順很多。

我覺得好的技術溝通，不是把內容講得很技術，而是把複雜問題整理成不同角色都能理解、討論，並且做出下一步決策的形式。
