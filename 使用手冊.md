# Blog 使用手冊

這份文件是給人看的操作手冊。AI agent 的工作規則放在 `AGENTS.md`，這份文件則說明你平常要怎麼寫文章、檢查 SEO、產生內部連結建議，以及調整網站設定。

## 這個 Blog 是怎麼運作的

這個網站使用 MkDocs Material 建立。

簡單流程是：

```text
Markdown 文章
  -> MkDocs 讀取 mkdocs.yml
  -> 產生 site/ 靜態網站
  -> 部署到 GitHub Pages / Cloudflare Pages / Netlify 等平台
```

文章本體都放在：

```text
docs/writing/posts/
```

網站設定主要在：

```text
mkdocs.yml
```

本專案刻意把文章輔助工具寫成 JavaScript，放在：

```text
scripts/
```

## 第一次使用

如果 `.venv` 已經存在，通常不用重做安裝。若要重新安裝 Python/MkDocs 依賴：

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

確認網站可以 build：

```bash
npm run build
```

啟動本機預覽：

```bash
npm run dev
```

預設網址是：

```text
http://127.0.0.1:8000/
```

## 建立新文章

使用這個指令建立新文章：

```bash
npm run post:new -- "你的文章標題"
```

它會在 `docs/writing/posts/` 下面建立一個 Markdown 檔案，例如：

```text
docs/writing/posts/2026-05-24-my-post-title.md
```

新文章預設會是 `draft: true`，表示草稿。文章準備發布時再改成：

```yaml
draft: false
```

## 文章格式

每篇文章最上方都要有 frontmatter，也就是用 `---` 包起來的 metadata。

基本格式：

```yaml
---
title: "文章標題"
date: 2026-05-24
description: "一段 50 到 160 字左右的 SEO 摘要，說清楚這篇文章在講什麼。"
categories:
  - Notes
tags:
  - blog
  - setup
authors:
  - you
draft: false
---
```

接著才是正文：

```markdown
# 文章標題

這裡寫開頭。第一段最好直接說明這篇文章的主題、問題或核心觀點。

<!-- more -->

## 第一個段落

正文內容。

## 第二個段落

更多內容。
```

## Frontmatter 欄位說明

`title`：文章標題，會用在頁面標題、文章列表和 SEO。

`date`：發布日期，格式固定為 `YYYY-MM-DD`。

`description`：SEO 摘要，建議 50 到 160 字。這會影響搜尋結果、RSS 和文章預覽。

`categories`：文章分類。第一版建議一篇文章只放一個主要分類，例如 `Notes`、`AI`、`Software Engineering`。

`tags`：文章標籤，可以放多個，用來描述更細的主題。

`authors`：作者，目前預設是 `you`，對應 `docs/writing/.authors.yml`。

`draft`：是否為草稿。`true` 表示草稿，`false` 表示準備發布。

## 寫文章的建議規則

每篇文章建議遵守：

- 只使用一個 H1，也就是 `# 文章標題`
- 主要段落用 H2，也就是 `## 段落標題`
- 小段落用 H3，也就是 `### 小段落標題`
- 開頭先講清楚文章要解決什麼問題
- 在開頭摘要後放 `<!-- more -->`
- 連結文字要有意義，不要寫「點這裡」
- 發布前確認 `description` 不是空的
- 發布前確認 `draft: false`

## SEO 檢查

每次新增或修改文章後，建議先跑：

```bash
npm run seo:check
```

它會檢查：

- 是否缺少 `title`
- 是否缺少 `date`
- 是否缺少 `description`
- 是否缺少 `categories`
- 是否缺少 `tags`
- 是否缺少 `authors`
- `description` 長度是否合理
- 日期格式是否正確
- 檔名 slug 是否和標題大致一致

如果缺 description，可以先用本地摘要工具產生草稿：

```bash
npm run desc:generate
```

確認輸出看起來合理後，再寫入檔案：

```bash
npm run desc:generate -- --write
```

注意：這個工具目前不是 AI，只是用文章第一段產生簡單摘要。重要文章建議你自己改成更好的 human-written description。

## 內部連結與 Crosslink

這個專案參考 `jxnl/blog` 的做法，會產生一份給 AI 或工具讀的文章地圖：

```bash
npm run content-map:generate
```

它會產生：

```text
content-map.json
```

裡面包含每篇文章的：

- title
- slug
- date
- categories
- tags
- summary
- 主要關鍵詞
- content hash

接著可以跑：

```bash
npm run crosslink:suggest
```

它會根據文章之間的主題、tag、category 和關鍵詞，給你內部連結候選。

目前文章很少時，可能會顯示沒有建議，這是正常的。等文章累積到 3 到 5 篇以上，這個功能才會比較有用。

內部連結建議規則：

- 一篇文章通常放 2 到 5 個高品質內部連結就好
- 不要因為有關鍵字就硬連
- 連結應該幫助讀者理解下一步
- 使用描述性 anchor text
- 連到其他文章時使用相對連結，例如：

```markdown
我在 [第一篇文章](./2026-05-24-first-post.md) 裡整理過這個 workflow。
```

## Sitemap 與 RSS

產生 XML sitemap：

```bash
npm run sitemap:generate
```

這會寫入：

```text
docs/sitemap.xml
```

RSS 由 MkDocs plugin 在 build 時處理。

發布前建議跑：

```bash
npm run sitemap:generate
npm run build
```

## 常用指令

```bash
# 啟動本機預覽
npm run dev

# 建立新文章
npm run post:new -- "文章標題"

# 檢查文章 SEO metadata
npm run seo:check

# 產生 missing description 的建議
npm run desc:generate

# 寫入 missing description
npm run desc:generate -- --write

# 產生 AI/工具可讀的文章地圖
npm run content-map:generate

# 產生內部連結建議
npm run crosslink:suggest

# 產生 sitemap.xml
npm run sitemap:generate

# build 靜態網站
npm run build
```

## 發布前 Checklist

發布文章前建議照這個順序：

```bash
npm run seo:check
npm run content-map:generate
npm run crosslink:suggest
npm run sitemap:generate
npm run build
```

手動檢查：

- `draft` 是否已改成 `false`
- `description` 是否自然、準確
- `categories` 和 `tags` 是否合理
- `<!-- more -->` 是否放在開頭摘要後
- 內部連結是否真的有幫助
- 本機預覽文章頁是否正常

## 網站設定要改哪裡

主要設定檔是：

```text
mkdocs.yml
```

常見需要改的地方：

`site_name`：網站名稱。

`site_description`：整個網站的 SEO 描述。

`site_url`：正式部署後的網站網址。這很重要，會影響 sitemap、canonical URL 和 RSS。

`repo_url` / `repo_name`：GitHub repo 連結。

`theme`：MkDocs Material 主題設定。

`nav`：上方或側邊導覽。

`plugins`：搜尋、blog、RSS、minify 等功能。

`extra.social`：社群連結。

作者設定在：

```text
docs/writing/.authors.yml
```

如果你要把作者從 `you` 改成你的名字，就改這裡，也要同步調整文章 frontmatter 的 `authors`。

## 視覺樣式要改哪裡

目前主要樣式檔：

```text
docs/stylesheets/extra.css
```

比較大的 MkDocs theme 客製化放在：

```text
overrides/
```

如果只是改字體大小、顏色、文章寬度、連結樣式，通常改 `extra.css` 就夠。

如果要改文章列表、header、footer、layout，才會需要動 `overrides/`。

## AI 協作規則

給 AI agent 看的規則放在：

```text
AGENTS.md
CLAUDE.md
.cursor/rules/
.claude/agents/
```

你平常不一定要看這些檔案。它們的用途是讓 Codex、Claude 或 Cursor 知道：

- 文章放在哪裡
- frontmatter 要有哪些欄位
- SEO 要怎麼檢查
- crosslink 要怎麼做
- 不要隨便改設計
- 不要隨便改已發布文章的 URL

## 目前架構的限制

這個網站很適合：

- 個人 blog
- 技術文章
- SEO 友善文章站
- Markdown-based 內容管理
- AI 輔助寫作與內部連結

但它不適合：

- 會員系統
- 複雜互動 app
- 大量 React component
- 很自由的 landing page 動畫
- 需要資料庫的功能

如果未來你想要非常客製化的 UI，可以考慮搬到 Astro 或 Next.js。不過目前文章都用 Markdown 和乾淨 metadata 管理，所以未來搬家不會太痛。
